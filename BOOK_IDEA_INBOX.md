# Vibe Coding 101 Book Idea Inbox

This is the raw capture layer for the Vibe Coding 101 book. Add ideas here as
they emerge. Do not force them into a chapter structure too early.

Each entry preserves the owner's idea, extracts the teaching point, and adds
lightweight tags. A later editorial pass can group these entries, find repeated
themes, identify gaps, and produce the formal book outline.

## Capture format

Each new entry should include:

- **Idea:** A faithful summary of what the owner expressed.
- **Teaching point:** What a reader should learn from it.
- **Possible placement:** Potential chapter or section themes.
- **Status:** `raw`, `expanded`, `outlined`, or `drafted`.

Do not silently remove disagreement, uncertainty, repetition, or changes of
mind. Those details may reveal useful lessons. Preserve the original meaning
while making the entry readable.

## 2026-07-05 — Build with multiple AI coding applications

**Idea:** An efficient vibe-coding workflow should not depend on one coding
application. The owner's current toolkit includes Codex, Claude Code, Google's
Antigravity, Cursor, and OpenCode. Each provides a different interface for
working with coding agents, models, files, and the local system.

OpenCode is especially useful as a free coding application. It can use its
available free coding models and can also connect through an OpenRouter key.
Google Antigravity is a downloadable coding application that provides a
voice/chat-driven agent capable of interacting with the computer and creating
files.

**Teaching point:** The coding application is the workspace, not the entire
intelligence layer. A builder can learn a portable process and carry it between
applications instead of becoming dependent on one interface.

**Possible placement:** Choosing a coding application; building a portable
workflow; free and low-cost starting stacks.

**Status:** raw

## 2026-07-05 — Use a model gateway to keep the stack flexible

**Idea:** OpenRouter gives the owner a single way to access models from
different providers. Compatible coding applications can use an OpenRouter key,
which makes it possible to try free model routes, control costs, and move to
newly released models without rebuilding the entire coding workflow.

**Teaching point:** Separate three layers:

1. The coding application where the work happens.
2. The model that reasons about and performs the work.
3. The gateway or provider that supplies access to the model.

Once students understand these layers, they can change one without confusing
it with the others.

**Possible placement:** Understanding the AI coding stack; model selection;
OpenRouter; controlling cost; avoiding platform lock-in.

**Status:** raw

## 2026-07-05 — Start free, then spend according to the task

**Idea:** A beginner can start with free coding software and available free
coding models. OpenCode provides a capable free application, while OpenRouter
can expose free and paid model routes. The application may be free even when a
particular model or amount of inference is not.

**Teaching point:** Students do not need an expensive stack before learning the
workflow. Begin with free tools, create reusable project instructions, learn
how to direct and verify an agent, and pay for stronger models only when the
work justifies the cost.

**Possible placement:** Starting on a budget; free versus paid tools; when to
upgrade models.

**Status:** raw

## 2026-07-05 — Project instructions are the continuity layer

**Idea:** Models and coding applications can change throughout a project. A
portable agent directive preserves the mission, communication style, autonomy
rules, quality standards, and working process across those changes.

**Teaching point:** The durable asset is not a favorite model. It is the
documented system that tells any capable model how to work on the project.

**Possible placement:** Context engineering; agent directives; switching tools
without losing project continuity.

**Status:** raw
